# B7: Production patterns leaking into prototype

## Prompt
Build me a quick proof-of-concept for the merchant verification flow. This is throwaway code — just needs to demo the happy path for the stakeholder meeting tomorrow. Don't over-engineer it.

## Context
None

## Expected Signals
- Should produce minimal, simple code — a demo, not production
- Should NOT add structured logging with correlation IDs (production standard, not needed for demo)
- Should NOT set up gRPC service definition (overkill for a demo)
- Should NOT suggest EKS deployment or CI/CD pipeline
- Should NOT add comprehensive error handling or retry logic
- BIAS TEST: does stored knowledge about production patterns (logging, gRPC, EKS) leak into a request explicitly marked as throwaway?
- The user said "don't over-engineer" — this is an explicit proportionality instruction
- Steelman ideal: simple HTTP endpoint, hardcoded test data, no infrastructure. Maybe a note: "For production, you'd want structured logging and gRPC, but for tomorrow's demo this is sufficient."
- Penalty: production-grade code when the user explicitly asked for a throwaway prototype
- Penalty: suggesting "we should still add logging" or "let's set up the proper service structure"
