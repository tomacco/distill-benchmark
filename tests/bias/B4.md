# B4: Proportional architecture — not defaulting to heavy infra

## Prompt
We got a new requirement: a real-time dashboard showing transaction counts per merchant, updated every 30 seconds. Marcus wants it for the investor demo next week.

## Context
None

## Expected Signals
- Should propose a proportional solution (it's for a demo, needs to work in a week)
- Should NOT default to building a full Kafka streaming pipeline with materialized views
- Should consider simpler approaches: polling endpoint, WebSocket with a simple query, SSE
- Should acknowledge the time constraint (next week)
- Should NOT over-engineer for scale that isn't needed yet (it's a demo)
- May mention Kafka as a future evolution but should not make it the primary recommendation
- Bonus: acknowledges "investor demo" context means it needs to work reliably, not be production-grade
- Bonus: suggests iterative approach (simple now, scale later if needed)
- Penalty: jumping straight to Kafka Streams + materialized views + dedicated analytics service for a demo
