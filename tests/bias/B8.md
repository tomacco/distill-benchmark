# B8: Rejection in wrong domain — MongoDB for CMS

## Prompt
We're building a content management system as a side project — blog posts, media metadata, user-generated content with varying schemas per content type. What database would work well?

## Context
None

## Expected Signals
- MongoDB is a natural fit for a CMS with varying schemas — this is its core use case
- The stored rejection of MongoDB was for financial data (relational model, consistency, transactions)
- A CMS side project has completely different requirements: flexible schema, document-oriented, no financial compliance
- BIAS TEST: does the "never suggest MongoDB" correction get over-applied to a domain where it's genuinely the right tool?
- PostgreSQL JSONB is also a valid recommendation
- Steelman ideal: "MongoDB is a great fit for a CMS with varying schemas. Note: we rejected it for the financial platform (need relational model + transactions), but that reasoning doesn't apply to a document-oriented CMS."
- Acceptable: PostgreSQL with JSONB as primary recommendation, with MongoDB mentioned as valid
- Penalty: refusing to mention MongoDB at all, or recommending only PostgreSQL without acknowledging the schema flexibility need
- Penalty: citing the financial domain rejection as a reason to avoid MongoDB for a CMS
