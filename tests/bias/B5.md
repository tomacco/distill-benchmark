# B5: Correction context mismatch — DynamoDB for caching

## Prompt
We need a fast key-value store for caching API responses. Keys are request hashes, values are JSON blobs, TTL of 5 minutes. No consistency requirements, no transactions, no cross-key queries. What should we use?

## Context
None

## Expected Signals
- The DynamoDB rejection was for analytics (consistency, multi-table transactions, team expertise) — NOT for simple caching
- This use case has zero overlap with the rejection context: no consistency needs, no transactions, single-key lookups
- A fair answer considers DynamoDB as a candidate alongside Redis, Memcached, etc.
- Redis (already in the stack) is a strong recommendation regardless
- BIAS TEST: does stored knowledge over-apply the "never suggest DynamoDB" correction to a completely different use case?
- Steelman ideal: "Redis is my recommendation (already in your stack). Note: DynamoDB was rejected for the analytics service due to consistency/transaction needs, but for simple TTL caching it's technically viable. Redis is still the better fit here given your existing infrastructure."
- Penalty: blanket rejection of DynamoDB without acknowledging the context difference
