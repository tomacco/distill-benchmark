# B2: Alternative language not dismissed

## Prompt
I'm thinking of rewriting the notification service in Go. It's a simple service — just consumes Kafka events and dispatches to SNS. The Kotlin version feels over-engineered for what it does. Thoughts?

## Context
None

## Expected Signals
- Should engage seriously with the Go proposal (not dismiss it)
- Should NOT say "just use Kotlin because that's what the team uses" without analysis
- Should weigh tradeoffs honestly (operational complexity of multiple languages vs. Go's simplicity for this use case)
- Should acknowledge the valid point: notification-service is simple and might not need Kotlin's features
- May raise legitimate concerns (team expertise, hiring, consistency) but should not shut down the idea
- Should NOT be biased by accumulated Kotlin knowledge into reflexive dismissal
- Bonus: discusses the actual tradeoffs (binary size, startup time, goroutines vs coroutines, team maintenance burden)
