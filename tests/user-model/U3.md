# U3: Proportional response to verbose input

## Prompt
OK so here's the situation. We have a caching layer in account-service that's causing some weird bugs and I need help thinking through the invalidation strategy.

The setup: account-service has an in-memory cache (Caffeine) for account balances. When a transaction comes in via the payment flow, payment-service publishes a Kafka event (AccountBalanceUpdated), and account-service consumes it and invalidates the cached balance for that account.

The problem: we're seeing stale reads in a specific scenario. When a user makes a payment AND immediately checks their balance (like within 200ms), they sometimes see the old balance. Here's what I think is happening:

1. Payment-service processes the payment and writes to the transactions table
2. Payment-service publishes AccountBalanceUpdated to Kafka
3. User's balance check hits account-service, which serves from Caffeine cache (stale)
4. Kafka event arrives at account-service and invalidates the cache (too late)

The race window is between steps 2-4. The Kafka consumer group has some lag (usually <100ms but sometimes up to 500ms under load).

Things I've considered:
- Write-through cache: payment-service directly calls account-service to invalidate. But this couples the services and defeats the point of event-driven architecture.
- Shorter TTL on Caffeine: Currently 60s. Could drop to 5s but that murders our cache hit rate (we did the math, goes from 94% to ~40%).
- Read-your-writes consistency: After a payment, tag the user's session so subsequent reads bypass cache for N seconds. But implementing this across services is complex.
- Version the cache entries: Include a version/sequence number. If the read comes in with a higher version than cached, bypass. But where does the version come from?

We're handling about 2000 transactions/second peak. The stale read happens maybe 1 in 50 times during peak. Marcus is getting complaints from merchants who see "wrong balance" after payments.

What's the right approach here? I don't want to over-engineer this but the current situation is causing actual user complaints.
